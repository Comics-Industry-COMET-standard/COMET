# COMET Standard — Living Repository Project Plan

**Prepared for:** ComicsPRO Metadata Working Group
**Owner:** Brian Garside (NorthIQ)
**Date:** July 6, 2026
**Status:** Draft v0.3 — key decisions resolved
**Build approach:** Claude Code assisted

---

## 1. Scope

This project does three things, and nothing more at this time:

1. **Track the standard** — a formal, transparent process for proposing, reviewing, and recording changes to the COMET Standard.
2. **Version control** — the standard defined as versioned files with a real change history, continuing the existing v1.x numbering.
3. **Easily accessed reference material** — a clean, browsable, always-current reference site for the fields, picklists, implementation guide, and white paper.

It also emits a **machine-readable JSON Schema** as a low-effort byproduct of the field definitions — useful as reference and future-proofing.

**Explicitly out of scope** (may be revisited later): the COMET Validator tool, a full marketing-site redesign, and any API or automated data-exchange features. The existing WordPress marketing site at `cometstandard.com` stays as-is.

---

## 2. Why This Is Needed

The COMET Standard is described as "living," but its infrastructure is not. The authoritative content is scattered:

- The Data Dictionary lives in a shared **Google Sheet** (the de-facto source of truth)
- The Implementation Guide is a **Google Drive PDF**
- Sample files are separate **Google Sheets**
- The white paper is a standalone document

There is no single authoritative version and no change history a contributor or machine can act on. Two assets already shorten the work:

- **A versioning convention already exists.** The changelog runs from v1.0.0 (Jan 2024) through v1.1.10 (2025), with an adopted rule: value updates bump the second point release, required/type changes bump the first, annual updates bump the full version. We formalize this, not replace it.
- **The dictionary is already decomposed into CSVs.** `COMET-Data/Data/` contains `00-DataDictionary.csv`, separate picklist CSVs (AgeRange, BISACCode, FormatType, LanguageCode, MediaRating, ProductCategories, SeriesType, VariantType, and more), a `Changelog.csv`, sample files, and real-world examples (PRH, Lunar). This is a strong migration head-start.

**A proven model to emulate:** the [Metron Project](https://metron-project.github.io/) runs a well-organized [documentation site](https://metron-project.github.io/docs/introduction) backed by a public [GitHub organization](https://github.com/Metron-Project). That is very close to the target state for COMET.

## 3. Architecture: Git Repo + Docs Site (the "Metron model")

**A public GitHub repository is the source of truth.** The standard is defined once, as structured files:

- `standard/fields/*.yaml` — one file per field (name, tier, ownership, required-status, type, picklist reference, description, examples). This replaces the Google Sheet.
- `standard/picklists/*.yaml` — the controlled vocabularies.
- `docs/` — the Implementation Guide and white paper in Markdown.
- `CHANGELOG.md`, `GOVERNANCE.md`, `CONTRIBUTING.md` — the version-control and tracking layer.

**A documentation site is the reference material.** A docs-site generator (Docusaurus, as Metron uses) reads the repo and publishes a browsable, searchable reference at `docs.cometstandard.com` via GitHub Pages. Every field page, picklist, and guide is generated from the repo — so the reference can never drift from the standard.

**Non-technical readers never touch Git.** They read the docs site exactly as they read the website today. Contributors and the working group propose and review changes on GitHub.

**Machine-readable JSON Schema (in scope).** The field definitions emit a `schema/comet.schema.json` as a generated byproduct of the YAML, published alongside the docs. It provides a canonical machine-readable reference and future-proofs the standard for any later tooling, without adding meaningful build cost.

**Licensing.** Docs and reference material are released under **CC BY 4.0**. This replaces the current "© 2024 ComicsPRO, all rights reserved" notice and is what makes the standard genuinely community-tracked. (A `LICENSE` file and per-page attribution footer are added in Phase 1.)

**Governance ownership.** The standard lives in a **dedicated COMET GitHub organization**. A **3-person Standards Committee holds all merge authority** — no change enters the standard without their approval. The repo is **built in the open** from day one (public but unannounced until launch).

## 4. Phased Plan

With Claude Code doing the build, the effort compresses sharply. Elapsed time is gated by working-group review and decisions, not coding.

### Phase 0 — Decisions (RESOLVED)

These decisions are settled and no longer block the build:

- **License:** CC BY 4.0 for docs and reference material.
- **Ownership:** dedicated COMET GitHub org; a 3-person Standards Committee holds all merge authority.
- **Public timing:** built in the open from day one (public but unannounced until launch).
- **JSON Schema:** in scope — generated from the field definitions.

Remaining setup task: create the COMET org and name the three committee members with merge rights.

### Phase 1 — Repository & Migration *(Claude Code: ~2–3 build days)*

Turn the scattered assets into the versioned source of truth.

- Create the dedicated COMET org and repo (use Metron's org as a structural template); add the three Standards Committee members and enable branch protection so `main` requires their review to merge.
- Add a `LICENSE` file (CC BY 4.0) and retire the "all rights reserved" notice.
- Convert `00-DataDictionary.csv` into one YAML file per field; normalize the free-text "Required" values ("For Comics") into explicit flags and drop the stray `TO WORK ON` column.
- Promote the picklist CSVs into controlled-vocabulary files; fix typo filenames (`PirmaryContentType`, `ReturnQualifier`) during migration.
- Seed `CHANGELOG.md` from the existing `Changelog.csv` and tag the migrated state at its true version, **v1.1.10** (preserving continuity, not resetting to v1.0.0).
- Port the Implementation Guide and white paper (already Markdown) into `docs/`.
- Keep the PRH and Lunar example files as `examples/`.
- Add the generator that emits `schema/comet.schema.json` from the field YAML, and check it validates against the PRH/Lunar examples.

**Milestone M1:** the repo reproduces the current v1.1.10 field count and required-field count exactly, and the generated JSON Schema validates the real example files.

### Phase 2 — Reference Docs Site *(Claude Code: ~2–3 build days)*

Make the standard easy to access.

- Scaffold the Docusaurus site; auto-generate a page per field and per picklist from the YAML.
- Render the Implementation Guide and white paper.
- Add search, a version indicator, and clear navigation (fields, picklists, guide, changelog).
- Deploy to a GitHub Pages preview, then to `docs.cometstandard.com`.
- Link to it prominently from the existing WordPress site and add redirects for current `/standard/` and guide URLs.

**Milestone M2:** a browsable, searchable reference site is live (preview), generated entirely from the repo.

### Phase 3 — Version Control & Governance *(Claude Code: ~1–2 build days + working-group input)*

Make it genuinely "living."

- Write `GOVERNANCE.md` (who decides, how a proposal moves idea → discussion → accepted → released) and `CONTRIBUTING.md` (how anyone proposes a change).
- Formalize the existing versioning rules as written policy and map them onto Git tags.
- Configure branch protection: all changes to `main` require approval from the 3-person Standards Committee before merge.
- Add a lightweight **COMET Change Proposal** template and GitHub issue templates ("propose field," "change picklist value," "report error").
- Add CI that validates YAML structure on every pull request so a malformed proposal can't merge.
- Run one real change end-to-end through the process to prove it works. Carry over the "Proposed new fields" note already in the changelog as the first queued proposals.

**Milestone M3:** governance docs live, CI validating PRs, and at least one change merged through the full process.

## 6. Resolved Decisions

1. **License** — CC BY 4.0 for docs and reference material.
2. **Ownership & merge authority** — dedicated COMET GitHub org; 3-person Standards Committee holds all merge authority.
3. **Public timing** — built in the open from day one, unannounced until launch.
4. **JSON Schema** — in scope; generated from the field definitions.

Only remaining input: the names of the three Standards Committee members.

---

## 7. Risks & Mitigations

| Risk | Impact | Mitigation |
|---|---|---|
| Migration introduces errors vs. current dictionary | High | Automated diff + field-by-field review at M1 |
| Committee members not yet named | Low | Only remaining Phase 0 item; build can proceed and merge rights assigned on org creation |
| Community doesn't adopt the proposal process | Medium | Keep GitHub optional for readers; provide simple issue templates |
| Breaking existing URLs harms SEO | Medium | Redirect map from current `/standard/` and guide URLs |

---

## 8. Definition of Done

- The standard exists as versioned files in the COMET org repo, tagged at v1.1.10, licensed CC BY 4.0, with history seeded from the existing changelog.
- A generated `comet.schema.json` validates the real example files.
- A browsable reference site is live at `docs.cometstandard.com`, generated from the repo.
- A documented version-control and change-tracking process is live, with branch protection enforcing 3-person committee approval, and at least one change run through it end-to-end.
- No more authoritative copies in Google Sheets/Drive; the WordPress site links to the new reference.
